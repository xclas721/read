# W3 課前作業 — 需求詳細清單(每人一份)

> 三組的衝突整理你都拿到了——綜整成你自己的一份結構化需求清單。這是 W3 你寫 spec 的原料,課堂上會直接長成一份完整 spec。
> 這份**等於 spec 的三段**:功能需求 / 非功能需求 / 開放問題。想直接用 [spec-template.md](spec-template.md) 填這三段也可以(其餘段留到課堂),效果一樣。
> 配合自學包 [m3-analysis.md](m3-analysis.md)(結構化、MoSCoW、需求 vs 解法)與 [m5-sdd.md](m5-sdd.md) 一起用。載體 Markdown,約 30–45 分鐘。

## 2. 非功能需求

> 效能 / 安全 / 可用性 / 合規。有就寫具體(數字、條款名);沒有標 N/A。
> ⚠️ 效能類**必附規模假設**(約幾筆資料、約幾 QPS / 多久一次)。寫不出規模的效能需求先別列——很可能就是上週那種「量化後就消失」的假議題。

- **效能**:[例:單筆查詢 < 1s。規模假設:客戶約 ____ 筆、請求型態 ____(觸發式 / 高併發)]
- **安全**:[例:個資(email / 手機 / 姓名)加密不明碼;查詢留 audit log]
- **可用性**:[例:工具掛掉要能用 ____ 備份立即查。規模假設:____]
- **合規**:[例:依個資法;ISO-27001 資料異動留紀錄]

---

## 3. 開放問題 / 待 escalate

> 沒拍板的、牽涉風險的、兩案並陳的——全列。每條標「需向誰確認」。藏起來,讀的人會以為你想清楚了。

- [ ] [例:批次查詢 / 人工審核流程要不要做?] — 待主管拍板
- [ ] [例:domain 自動歸戶 vs 人工防詐騙的資安風險] — 需先向主管講風險再決定
