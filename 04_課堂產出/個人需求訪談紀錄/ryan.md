# 三、目前需求中可能存在的衝突與風險

## 衝突 1：主管說第一階段只開放部門內使用，但問題中提到是否開放給業務

### 目前資訊

| 來源     | 說法                                   |
| -------- | -------------------------------------- |
| Ardi     | 第一階段只開放部門內使用，不開放業務   |
| 原始提問 | 曾討論是否可以開放給外部使用，例如業務 |

### 衝突 / 風險

目前結論是不開放業務，但未來業務很可能仍會有查詢需求，尤其若業務是客戶關係人或客訴來源之一。

風險是第一階段若完全沒考慮業務角色，後續擴充權限模型時可能需要重構。

### 建議解法

第一階段可以不開放業務，但權限模型不要寫死為「部門內 / 非部門內」二分法。

建議先設計可擴充 RBAC：

| 角色            | 第一階段是否啟用       |
| --------------- | ---------------------- |
| Admin           | 啟用                   |
| Editor          | 可考慮                 |
| Viewer          | 啟用                   |
| AI Agent        | 啟用但需 Approve       |
| Business Viewer | 暫不啟用，保留角色設計 |
| Auditor         | 可保留                 |

後續決策問題：

> 業務未來是否可能需要唯讀查詢？如果可能，第一階段至少要保留 Business Viewer 的設計空間。

## 衝突 2：人為修改不需要第二人審核，但 AI Agent 修改需要人工 Approve

### 目前資訊

| 來源 | 說法                                                       |
| ---- | ---------------------------------------------------------- |
| Ardi | 人為修改資料用權限區分與操作歷史控管，不一定需要另一人審核 |
| Ardi | AI Agent 可以操作資料，但相關操作需人為在後台 Approve      |

### 衝突 / 風險

這不一定是衝突，但有治理邏輯不一致的風險。

如果 AI Agent 修改要審核，但人為修改不用審核，可能出現：

- 人為誤改資料無需審核
- AI 操作反而比人類更嚴格
- 敏感欄位缺少一致控管
- 後續稽核時難以解釋資料治理規則

### 建議解法

不要用「人為 / AI」作為唯一審核標準，應改用「操作風險等級」決定是否需要 Approve。

建議規則：

| 操作類型                        | 是否需要 Approve             |
| ------------------------------- | ---------------------------- |
| 一般備註修改                    | 不需要，留操作紀錄           |
| 客戶聯絡窗口新增                | 可不需要，留操作紀錄         |
| 客戶職位 / 角色修改             | 可不需要或視角色決定         |
| 客戶標籤修改                    | 視標籤類型決定               |
| 重要客戶 / VIP / 高風險標籤修改 | 建議需要 Approve             |
| 刪除客戶資料                    | 建議需要 Approve             |
| 批次匯入覆蓋既有資料            | 建議需要 Approve             |
| AI Agent 任何寫入操作           | 第一階段建議全部需要 Approve |

後續決策問題：

> 是否要依照操作風險分級，而不是只依照操作者是人或 AI 來決定審核流程？

## 衝突 3：資料要定期確認與手動更新，但又要求資料準確性與回應時間

### 目前資訊

| 來源          | 說法                                                     |
| ------------- | -------------------------------------------------------- |
| Ardi          | 用排程定期詢問客戶資訊是否正確，若有異動再人工登打更新   |
| Diego         | 最重視資料準確性、回應時間                               |
| Diego / Wayne | Admin 後台資訊要同步到客戶資訊工具，但第一階段先手動更新 |

### 衝突 / 風險

手動更新通常會造成：

- 資料延遲
- 資料不一致
- 人工登打錯誤
- 回應時間受限於資料維護流程
- Admin 後台與客戶資訊工具可能不同步

這與「資料準確性、回應時間」有潛在衝突。

### 建議解法

第一階段可以手動更新，但必須補上資料新鮮度與可信度機制。

建議每筆關鍵資料都要有：

| 欄位                                | 用途                     |
| ----------------------------------- | ------------------------ |
| data_source                         | 資料來源                 |
| last_verified_at                    | 最後確認時間             |
| last_updated_at                     | 最後更新時間             |
| updated_by                          | 最後更新人               |
| confidence_score / confidence_label | 可信度                   |
| stale_flag                          | 是否可能過期             |
| verification_status                 | 未確認 / 已確認 / 待更新 |

建議 Summary 顯示：

> 客戶資料可信度：低  
> 原因：超過 90 天未確認 / 最近查無更新 / 來源為手動登打  
> 建議：請檢查舊有資料或建立新資料

後續決策問題：

> 客戶資料多久沒有確認就算過期？30 天、60 天、90 天，還是依客戶類型不同？

## 衝突 4：Admin 後台資訊要同步到客戶資訊工具，但第一階段不要自動同步

### 目前資訊

| 來源          | 說法                                       |
| ------------- | ------------------------------------------ |
| Diego / Wayne | Admin 後台資訊要同步到客戶資訊工具         |
| Diego / Wayne | 第一階段先以手動更新為主，不要兩邊自動更新 |

### 衝突 / 風險

「要同步」與「不要自動同步」不是完全衝突，但需要明確定義同步方式。

若沒有定義清楚，可能會出現：

- Admin 後台改了，客戶資訊工具沒更新
- 客戶資訊工具改了，Admin 後台沒更新
- 不知道哪邊是正確資料
- 人員誤以為兩邊資料已同步
- AI Agent 根據舊資料產生錯誤 Summary

### 建議解法

第一階段建議定義為：

> Admin 後台是主要資料維護入口，客戶資訊工具讀取 DB 中的資料；若 Admin 後台與客戶資訊工具是兩套系統，第一階段採人工同步，但必須顯示同步狀態。

建議補充欄位：

| 欄位            | 用途                 |
| --------------- | -------------------- |
| source_system   | 原始資料系統         |
| sync_method     | 手動 / 自動 / 匯入   |
| last_sync_at    | 最後同步時間         |
| sync_status     | 成功 / 失敗 / 待同步 |
| conflict_status | 是否有資料衝突       |

後續決策問題：

> 第一階段到底是「同一個 DB 給後台與工具共用」，還是「兩邊各有資料，需要人工同步」？

## 衝突 5：客戶問題歷史紀錄放 Developer Site，但 AI Summary 可能需要歷史脈絡

### 目前資訊

| 來源         | 說法                                                 |
| ------------ | ---------------------------------------------------- |
| Ardi         | 客戶問題歷史紀錄用另外一個 Developer Site 管理與呈現 |
| Ardi / Diego | AI Summary 需要協助判斷是否需要處理                  |
| Diego        | 回覆客戶可依分類使用 Template                        |

### 衝突 / 風險

若歷史問題紀錄不在客戶資訊工具內，AI Agent 可能缺少上下文，影響：

- 是否為重複問題
- 是否為已知問題
- 是否曾有同類客訴
- 是否已有 workaround
- 是否需要升級
- 是否只是正常告警

如果 AI Summary 要判斷「是否需要處理」，通常需要歷史資料輔助。

### 建議解法

第一階段不一定要把 Developer Site 的資料完整搬進客戶資訊工具，但至少要做到關聯與引用。

建議做法：

| 階段     | 做法                                               |
| -------- | -------------------------------------------------- |
| 第一階段 | 客戶資訊工具顯示 Developer Site 連結               |
| 第二階段 | 透過 API 查詢該客戶近期問題摘要                    |
| 第三階段 | AI Summary 納入歷史問題、已知問題、RCA、workaround |
| 第四階段 | 客戶資訊、問題歷史、告警、回覆建議整合             |

後續決策問題：

> AI Summary 判斷是否需要處理時，是否需要讀取 Developer Site 的歷史紀錄？

| Ardi  | 第一階段以 DB 為主                                |
| Diego | 客戶資訊不可用時，要有 Excel 定期備份最新客戶資料 |
| Ardi  | 後台需有 Before / After 操作紀錄                  |

### 衝突 / 風險

Excel 備份可以作為 fallback，但如果沒有治理，會產生：

- Excel 與 DB 不一致
- Excel 外流風險
- Excel 被手動修改但沒有操作紀錄
- 回復資料時不知道哪份是最新
- 權限控管弱於後台系統

### 建議解法

Excel 應定位為「唯讀備援資料」，不是第二個正式維護入口。

建議規則：

| 項目       | 建議                       |
| ---------- | -------------------------- |
| Excel 用途 | 系統不可用時的唯讀查詢備援 |
| 匯出頻率   | 每日或每週                 |
| 儲存位置   | 受權限控管的內部儲存空間   |
| 是否可修改 | 不建議直接修改             |
| 是否加密   | 建議加密                   |
| 是否遮罩   | 依資料敏感度決定           |
| 使用紀錄   | 建議紀錄下載或存取紀錄     |
| 回復來源   | 仍以 DB 為主               |

後續決策問題：

> Excel 備份是給誰用？是否包含完整客戶資訊？是否需要遮罩敏感欄位？

## 衝突 7：用 CLI 控制回應內容，但也需要後台方便人類閱讀

### 目前資訊

| 來源         | 說法                                      |
| ------------ | ----------------------------------------- |
| Ardi         | 依客戶類型的不同回應，應該用 CLI 控制     |
| Diego        | 用誰能呼叫哪些 CLI 切分權限               |
| Ardi / Wayne | 後台需要 SSO、Admin、Viewer，方便人類閱讀 |

### 衝突 / 風險

CLI 與後台 UI 的定位需要切清楚。

如果沒有明確邊界，可能造成：

- 同一功能 CLI 可以做，UI 也可以做，但權限不一致
- CLI 查到的資料與 UI 顯示不同
- CLI 可繞過後台 Approve 流程
- 操作紀錄分散
- 使用者不知道應該用 CLI 還是後台

### 建議解法

建議定義 CLI 與後台 UI 的職責分工：

| 功能              | CLI                    | 後台 UI      |
| ----------------- | ---------------------- | ------------ |
| 查詢客戶資訊      | 可支援                 | 可支援       |
| 產生 AI Summary   | 可支援                 | 可顯示       |
| 查看客戶資料      | 可支援摘要             | 支援完整閱讀 |
| 修改客戶資料      | 第一階段不建議直接生效 | 主要操作入口 |
| AI Agent 修改審核 | 不建議在 CLI 完成      | 後台 Approve |
| 權限管理          | 不建議                 | Admin 後台   |
| 操作紀錄查詢      | 可查摘要               | 完整呈現     |

後續決策問題：

> CLI 是否只能讀取與觸發 AI Summary？還是也能提交修改申請？

## 衝突 8：功能重要性用使用頻率排序，但半夜告警判斷可能是低頻高價值需求

### 目前資訊

| 來源          | 說法                                     |
| ------------- | ---------------------------------------- |
| Diego / Wayne | 功能重要性以使用頻率排序                 |
| Ardi          | 半夜被叫起來時，要能立即判斷是否需要處理 |
| Diego         | 最重視資料準確性、回應時間               |

### 衝突 / 風險

使用頻率高的功能不一定是業務價值最高的功能。

例如：

| 功能                   | 使用頻率 | 價值 |
| ---------------------- | -------: | ---: |
| 查客戶 ID              |       高 |   中 |
| 半夜告警判斷是否需處理 |       低 |   高 |
| VIP 客戶重大客訴提示   |       低 |   高 |
| 可信度過低提醒         |       中 |   高 |

如果只用使用頻率排序，可能會犧牲低頻但高風險、高價值的需求。

### 建議解法

功能優先順序不要只看頻率，建議用加權模型：

| 指標                     |      權重建議 |
| ------------------------ | ------------: |
| 使用頻率                 |           25% |
| 風險降低                 |           25% |
| 作業時間節省             |           20% |
| 對 AI Agent 自動化的幫助 |           20% |
| 開發成本                 | 10%，反向計分 |

後續決策問題：

> 第一階段功能排序要只看使用頻率，還是要同時納入風險與值班場景價值？

## 衝突 9：不限制客戶聯絡窗口數量，但資料需要準確且方便使用

### 目前資訊

| 來源          | 說法                                        |
| ------------- | ------------------------------------------- |
| Diego / Wayne | 客戶聯絡窗口不需要特別限制                  |
| Diego         | 重視資料準確性                              |
| Ardi          | AI Summary 需要顯示角色、職位、是否技術人員 |

### 衝突 / 風險

不限制聯絡窗口數量是合理的，但如果沒有排序與狀態管理，會造成：

- 聯絡人過多，不知道該找誰
- 離職窗口仍被顯示
- AI Summary 抓錯聯絡人
- 技術 / 非技術角色判斷不準
- 無法判斷主要窗口、備援窗口、決策窗口

### 建議解法

不限制數量，但需要聯絡窗口治理欄位：

| 欄位              | 用途                                      |
| ----------------- | ----------------------------------------- |
| contact_status    | active / inactive / unknown               |
| contact_role      | 技術 / 非技術 / 主管 / 採購 / 業務 / 其他 |
| priority          | 主要窗口 / 次要窗口 / 備援窗口            |
| responsibility    | 技術問題 / 合約 / 帳務 / 維運 / 升級處理  |
| last_verified_at  | 最後確認時間                              |
| preferred_channel | Discord / Email / Phone / Teams / Line    |

後續決策問題：

> 當同一客戶有多個窗口時，AI Summary 應優先顯示哪一個？
