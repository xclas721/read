# SA 需求訪談 Workshop 整理 — 客戶資訊工具

> 客戶資訊工具 — 主管 / SRE / 服務 Owner 三方需求彙整
> 日期：2026/06/02

---

## 一、訪談對象總覽

| 對象 | 角色 | 關注焦點 | 對工具的態度 |
|------|------|---------|-------------|
| **主管** | 策略決策者 | 工具要做什麼、做到什麼程度、KPI | 設範圍、定方向、訂優先 |
| **SRE** | 系統可靠度負責人 | 系統怎麼活著、合規、可觀測性 | **不管業務邏輯與客戶資料**，只負責系統運作 |
| **服務 Owner** | 客戶端最高責任者 | 客戶體驗、資料正確性、出事我扛 | 重度使用者，期待救命的工具 |

## 二、主管的需求

### 1.1 主管想做什麼（依主管的話整理）

**範圍與使用者**
- 只開放給維運，**不開放給業務**，僅運營使用（Q2）
- 工具是內部開發用，API QA 不在 scope

**核心使用情境**（Q4）
- 想透過 AI Agent 讀取客戶從各管道來的訊息：微信、email、whatsapp 等
- 系統要能知道**客戶是誰**、**有疑問的服務系統是哪個**
- 信件內容問題分析、歷史問題追蹤**由另一個工具處理**（Q15）
- 對外寄信由**郵件處理工具**負責（不在本工具範疇）

**必要客戶資訊**
- 必填：姓名、email、角色（Q3）
- 客戶使用的服務的 SLA（Q8）

**客戶資料模型**（Q17）
```
公司
├── 使用的服務（1:N，可客製化覆寫 SLA）
├── 人（聯絡人）：姓名 / 通訊軟體 ID / email / 職位標籤 / 聯絡管道
└── 公司內對接窗口（我方員工）：業務窗口 / 技術窗口
```
- 業務人員離職：可批次修改成另一個業務

**AI Agent 介接**
- AI 權限：只讀（Q12）
- 新增/修改**客戶資料**需人工 approve
- 比對失敗：放在 AI Agent 的「行動建議」區呈現（Q1）
- **處理信件時需要回傳內容模板**，根據問題類型與 skill CLI 來回應（Q13）；

**行動建議產出**（Q5）
- summary 一眼知道嚴重度
- 是否要立刻處理
- 給出處理步驟（可執行行動）

**資料驗證**（Q18）
- 產出最終結果（例如要寄什麼信給誰）等人放行
- 內含資料信心度（資料超過一季/半年未更新 → 信心度降）

**覆核機制**（Q19）
- AI 改客戶資料 → 要人工覆核
- 人改 → 不用（可修改的人本來就權限高）

**資料維護**
- 每季排程自動匯出 Excel 給客戶確認（Q6）
- 匯出 → 匯入回系統更新（Q16）
- 匯出時若有敏感資料要特別處理（Q20）
- 操作紀錄：可以有（Q7）

**資料來源**（Q9）
- 第一階段：DB only
- 第二階段：加入合約資料，用 **RAG 讀合約內容**（非客戶資料）

### 1.2 主管還沒釐清的部分

- [ ] **預算上限是多少**——SRE 多次提到要看成本決定架構，但主管沒給範圍
- [ ] **Bulk query 是否納入範圍**——Owner 想看「所有 ACS 客戶」
- [ ] **第二階段 RAG 讀合約**——怎麼避免幻想？讀完是給 AI 還是給人？

## 三、SRE 的需求

### 2.1 SRE 關注什麼（依 SRE 的話整理）

**SRE 的職責邊界**（重要釐清）
- **不管業務邏輯、不管客戶資料**
- 只負責搭建系統、確保系統運作正常
- 因此 SRE 對「AI 行為」「客戶資料正確性」「業務權限」沒有意見

**合規定位**（Q1）
- 客戶資訊工具**不涉及 PCI**，可放在 CDE 範圍外
- 公司接下來要做 **ISO 27001 合規**，可能影響：儲存需加密、顯示不一定
- **個資法**須考慮

**網路安全**（Q2、Q5）
- ISO 合規一定要做白名單
- 請求端 IP 白名單

**基礎建設**（Q3、Q4）
- 預計放 GCP + k8s，盡量使用容錯架構
- 多工具同地部署，方便資料互傳
- 跨區資料同步靠 DB 一致性（Q9）

**監控指標**（Q6、Q7）
- API 調用頻率
- API 調用成功率（例如<80% 告警）
- 回應 latency / response time
- API 成敗定義：HTTP 2xx

**AI Token 監控**（Q8）
- 需要第三方組件監控用量
- 超量告警並停服務
- 通知設計者

**告警自動化分三階段**（Q10，**注意：是 infra 層面，非 AI 寫客戶資料**）
1. Run step
2. 半自動需 approve
3. 全自動
- 涉合規時：基礎建設異動要記錄、有權限的人 approve

**日誌策略**（Q11、Q12）
- 5xx/4xx 只能初步判斷，後續看 AP 日誌
- 想要日誌 summary：API 日誌 + AP 日誌整合，以客戶 ID 關聯
- ELK / Logstash 依預算

**流量保護**（Q14）
- 大量請求時用 **queue 排隊**避免掉球
- 各個工具間可能需要優先級的排序
- 搭配監控

**SLA 衝突解法**（Q16）
- 多久沒人回 → AI 先回中性罐頭信
- 或多久沒回 → 轉真人
- 加上**客戶優先等級**定義誰先處理

**Plan B（成本受限）**（Q13）
- 客戶資訊工具不是主要系統，處理時間可拉長
- 單一節點故障時，又有成本考量，就要考慮**部分人工取代**
- 機器要開多大需壓測後決定

### 2.2 SRE 還沒釐清的部分

- [ ] **預算上限**——重複強調但主管沒給數字
- [ ] **SLO 具體數字**——可用性、延遲、識別正確率分別定多少
- [ ] **「處理時間可拉長」具體拉長到多少**——30 秒？1 分鐘？5 分鐘？
- [ ] **「部分人工取代」的具體流程**——掛掉時怎麼通知？由誰接手？
- [ ] **ISO 27001 加密範圍**——哪些欄位？儲存層？顯示層？
- [ ] **告警 runbook** 誰寫、誰維護
- [ ] **壓測計畫**——誰負責、何時做、驗收標準

## 四、服務 Owner 的需求

### 3.1 Owner 想要什麼（依 Owner 的話整理）

**對工具的核心期待**（Q2）
- 想知道**客戶 email**（排查完查 email 回覆給他）
- 想看 **ACS 所有客戶**（一群、不是一筆）
- 希望**回傳時間短**（不然我自己查就好）
- 希望**資料正確、被驗證過**（例如發卡行的域名）
- **不能幻想回答**——來源資料庫是唯一真實正確來源

**更新頻率**（Q1）
- 一季或半年更新一次

**標籤規劃**（Q3）
- 客戶使用哪些系統
- 使用者職責
- **不需要**內部註解標籤（一律正式禮貌統一回覆）

**權限**（Q4）
- 後台查功能權限控管
- CLI 權限
- **一層或兩層尚未決定**

**聯絡人**（Q5）
- 目前是誰寄就回誰
- 多的客戶大約 8–9 人，**不需要設上限**

**來信者可信度**（Q6、Q7）
- 多一個標記：客戶資訊可信度
- 不在名單內 → 標記請人工確認
- 可信的話可直接加入名單
- 目前人工校驗方式：查過去信件 CC 名單、通訊群組搜尋

**功能優先序**（Q8）
- 依使用頻率排序
- 「**查 email**」最常使用，優先開發

**系統掛掉時的應急**（Q9）
- 目前用 gitlab 維護 MD 檔案
- 可以每天或每週定期下載備份

**罐頭信分類**（Owner 額外提出）
- 公告信、交易流程有固定回信 pattern
- 其他就沒有固定的
- 建議分類歸納成模板讓 AI 後續處理
- 可能屬於「交易 Summary 工具」或新增的「歷史信件分析工具」

### 3.2 Owner 還沒釐清的部分

- [ ] **Bulk query 還想看哪些列表**——除了所有 ACS，還有 VIP？特定機房？未活躍客戶？
- [ ] **權限一層 vs 兩層**還沒決定
- [ ] **系統掛掉時，如何在 SLA 內回應**——只有 gitlab 備份，但實際應急流程？
- [ ] **罐頭信模板庫的歸屬**——這工具？交易 Summary 工具？歷史信件分析工具？
- [ ] **「人工確認」誰來做**——Owner 自己？維運？什麼權限可以加入名單？

## 五、跨方衝突盤點（A vs B + 各自要問的問題）

### 🔴 衝突 1：Owner 要又快又準 vs SRE 在成本約束下用 queue 排隊

**A = Owner 說：**
- 「希望回傳時間短，不然我自己查就好」
- 「資料要正確、不能幻想」

**B = SRE 說：**
- 「大量請求時用 queue 避免掉球」
- 「客戶資訊工具不是主要系統，處理時間可拉長」
- 「單點故障 + 成本考量 → 部分人工取代」

**衝突本質：** 在大量請求 + 成本受限時，SRE 用 queue 換取系統不爆，這跟 Owner「我查一筆要立刻有結果」直接衝突。

#### 下一輪要問 Owner（A）的問題

- 「短」是多短？**2 秒、10 秒、1 分鐘、5 分鐘**——劃出底線
- 哪些查詢**絕對不能慢**？哪些可以接受稍慢？
- VIP 客戶 vs 一般客戶可不可以**不同延遲標準**？
- 排查緊急時，能不能接受「**先給部分答案，後續補完**」？
- 如果**回應慢但保證準確**，跟「回應快但偶爾不準」，你選哪個？

#### 下一輪要問 SRE（B）的問題

- queue 排隊期間的**平均/最壞延遲**是多少？有沒有實測值？
- 「處理時間可拉長」具體**拉長到多少**？
- 有沒有**不用 queue 的替代方案**（限流、cache、auto-scale）？
- **客戶優先等級** queue 怎麼實作？VIP 是不是可以完全跳過 queue？
- **預算上限是多少**？這直接決定 queue 是否必要

#### 如何收斂到對齊

```
Step 1: 先找主管 1:1 → 確認預算上限
Step 2: 帶預算數字給 SRE → 反推架構是否真需要 queue
Step 3: 把方案帶回 Owner → 確認可接受
Step 4: 三方對齊 → 列入 V1 設計
```

- 若預算足夠：可能根本不用 queue
- 若預算不足：用 **priority queue + VIP 不走 queue + 一般客戶接受 N 秒延遲**

### 🔴 衝突 2：使用粒度（單筆 vs 批次）

**A = 主管說：**
- 整個討論預設**單筆查詢**（一封信進來 → 識別一個客戶）

**B = Owner 說：**
- 「想知道 **ACS 所有客戶**」——明確的批次/分群需求

**衝突本質：** 查詢介面與資料模型可能要重新設計支援 bulk operation，主管沒有意識到這個需求存在。

#### 下一輪要問 主管（A）的問題

- 你知道 Owner 有 **bulk query 需求**嗎？是否同意納入範圍？
- bulk query 算 **MVP 還是 V2**？
- 如果納入 MVP，能不能調整工程資源/時程？
- 預算夠不夠多做 bulk query 功能？

#### 下一輪要問 Owner（B）的問題

- 除了「所有 ACS 客戶」，**還想看哪些列表**？
  - 所有 VIP？
  - 所有特定機房的客戶？
  - 所有近期沒互動的客戶？
  - 所有用某個服務的客戶？
- 拿到列表**接下來做什麼**？匯出？群發？分析？
- 多久要看一次？**每天、每週、臨時需要時**？
- 列表呈現需要哪些欄位？

#### 如何收斂到對齊

```
Step 1: Owner 先列出具體 bulk use cases（含數量、頻率）
Step 2: 帶清單給主管裁決：哪些納入 MVP、哪些 V2、哪些不做
Step 3: 工程估算成本，主管確認預算
```

### 🟡 衝突 3：權限分層具體規格未定

**A = 主管 Q19 暗示：**
- 「**可以修改的人就是權限高的人**」→ 兩層（檢視 vs 修改）

**B = Owner Q4 說：**
- 「後台查功能權限控管、CLI 權限——**還沒決定一層或兩層**」

**衝突本質：** 兩位都沒明確定義，但暗示的方向略有差異——主管假設兩層，Owner 公開承認還沒想。

#### 下一輪要問 主管的問題

- 「可以修改的權限高的人」**指的是哪些角色**？（特定維運？主管自己？）
- 一般檢視者**是哪些人**？所有維運？包括輪值人員？
- 兩層之外**需不需要更細**？例如：唯讀、編輯特定欄位、編輯所有、管理員

#### 下一輪要問 Owner 的問題

- 一層 vs 兩層的**考量點**是什麼？怕複雜？怕權限不夠細？
- 列出實際會使用工具的**具體角色清單**
- 每個角色**最少需要什麼權限**才能做事？**最多不該有什麼權限**？

#### 如何收斂到對齊

```
Step 1: Owner 列具體角色清單
Step 2: 三方一起做 RBAC 矩陣（角色 × 欄位/動作 = 權限）
Step 3: 確認最簡夠用的層級數
```

### 🟡 衝突 4：客戶資料更新責任（定期 vs 事件）

**A = 主管 Q6 說：**
- 排程**每季匯出 Excel** 給客戶確認（定期觸發）

**B = Owner Q6 說：**
- 客戶**不在名單時請人工確認**（事件觸發）

**衝突本質：** 兩個機制其實互補，但「誰負責執行」、「失敗怎麼處理」沒對齊。

#### 下一輪要問 主管的問題

- 每季匯出**由誰負責執行**（維運？業務？Owner？）
- 寄給客戶後**誰追蹤回覆**？
- 客戶**不回怎麼處理**——停權？降信心度？放著？
- 客戶**回的 Excel 格式亂掉**怎麼處理？

#### 下一輪要問 Owner 的問題

- 「人工確認」是 **Owner 自己做還是維運做**？
- 確認可信後**加入名單的權限**誰有？
- 兩種機制（定期 + 事件）**發現的資料修改**，要不要分流到不同流程？
- 你比較傾向哪一種機制當主、哪一種當輔？

#### 如何收斂到對齊

```
Step 1: 兩種機制併行（定期 + 事件雙觸發）
Step 2: 釐清各自的 owner、SLA、失敗處理
Step 3: 寫進 spec
```

## 七、下一堂課建議

### 推薦的訪談順序

從衝突盤點看，**先單獨找主管**是最有效率的——因為主管要解三件事：

1. **預算上限**（解開衝突 1、影響衝突 2）
2. **業務資料登打流程**（解開衝突 3 的內部矛盾）
3. **bulk query 是否納入範圍**（解開衝突 2）

這三題解開後，再單獨找 Owner 把細節釘住，最後三方對齊。

### 建議的下一場訪談對象

主管 / SRE / Owner 三方都訪談過了，但**還缺一塊核心拼圖：日常實際處理信件的運營人員**（Owner 底下那批人）。

這些人才是工具的**真正每日使用者**。建議下一堂安排這場訪談，重點在：

- 請他們**現場演一次**現在的工作流（不要抽象描述）
- 個人小抄、影子系統長什麼樣
- 信任邊界（AI 能做到哪一步才願意放手）
- 信件分類、罐頭信模板的真實使用情境
- 哪些 case AI 絕對不能自動處理
