# 客戶資訊工具 — 需求整理

---

## 一、主管角色

### 核心定位

這是一套供 AI Agent 呼叫的客戶資訊查詢工具，能根據呼叫者的標籤（身份類型）回傳對應資料，並具備智慧分流機制。期望效果是收到訊息或 email 時，能整理出 summary 讓使用者快速掌握當下所需的客戶資訊。未來若能進一步結合其他工具，則可協助釐清問題原因或直接完成後續處理步驟。

### 智慧分流邏輯

系統會先判斷問題性質與客戶類型，再決定處理方式。初期會先經真人確認後再送出，待精準度提升後可切換為自動回覆。業務角色方面，系統依業務人員的權限範圍回傳對應資訊，目前與客戶共用同一套系統，以權限區分可存取的內容與操作範圍。

### 後台管理介面

主管需求是提供可視化的維護介面，供管理員設定與調整標籤定義與對應資訊、分流規則，以及精準度門檻（決定何時從確認模式升級為自動模式）。資料更新頻率待擬定，需定期發訊息詢問客戶資訊並同步更新。

### 權限架構

目前僅供內部人員使用，可存取完整後台設定。內部工具目前尚無明確適用情境，暫不擴充，目前設定為 AI Agent 使用。其餘人員可能透過外部通知（如 Email、LINE、Slack）請求至 AI Agent 處理。

### 未來獨立專案

供單系統負責記錄每位客戶的歷史事件與歷史問題，建立完整的客戶脈絡。客服系統則透過 API 自動回覆客戶需求內容，並處理重複性問題。兩者皆另行建立新專案處理。

## 二、SRE

### 合規要求

系統需符合 PCI 與 ISO 規範。主要做法是不儲存敏感資料和 token 相關資料。

### 存取控管

目前工作業務範疇只需要客戶的白名單資訊。白名單的管理方式待確認是由內部人員手動維護。

### 監控需求

系統需監控以下 API 指標，以確保服務品質與可觀測性。監控項目包含 API 請求頻率、請求成功率（HTTP 200 回應）以及回應時間。

### 基礎架構說明

初期架構設計暫不考慮備援機制。

### 未來擴充方向

期望未來 AI 收到監控告警後，能自動啟動後續處理流程，實現告警的自動化應對。

## 三、OWNER

### 核心定位

這是一套供 AI Agent 呼叫的客戶資訊查詢工具，能根據呼叫者的標籤（身份類型）回傳對應資料，並具備智慧分流機制。期望效果是收到訊息或 email 時，能整理出 summary 讓使用者快速掌握當下所需的客戶資訊。未來若能進一步結合其他工具，則可協助釐清問題原因或直接完成後續處理步驟。

### 智慧分流邏輯

系統會先判斷問題性質，再決定是否需要調用資料。若為不需調用資料的告警，則先行過濾不處理。判斷邏輯分為以下幾個層次：

第一層為問題性質判斷，區分是否需要查詢客戶資料。若問題屬於一般性通知或不涉及客戶資訊的告警，則直接過濾，不進入後續流程。

第二層為客戶類型判斷，系統依據呼叫者的標籤識別身份，回傳對應權限範圍內的資料。

第三層為處理方式決策，初期由真人確認後再送出回覆，待系統精準度提升至一定門檻後，可切換為自動回覆模式。

### 後台管理介面

可沿用目前 oa-system 提供的可視化維護介面，供管理員進行標籤定義、分流規則與資料內容的設定與調整。資料更新頻率待擬定，目前預計以固定週期（一季或半年）主動發訊息詢問客戶，確認後同步更新資料。後台介面需支援以下操作：標籤新增與編輯、對應資訊的維護、分流規則的啟用與停用，以及資料更新紀錄的查閱。

### 基礎架構說明

為避免資料獲取失敗，目前暫定以固定時間排程下載檔案的方式作為備用資料來源，確保系統在無法即時存取資料時仍可正常運作。排程頻率與檔案格式待確認，下載後的資料需經過驗證與轉換，再寫入系統供查詢使用。資料版本管理與覆蓋機制亦需納入設計考量。

### 權限架構

目前仍在評估兩種方式。第一種是由 AI Agent 辨識後直接給予對應標籤，優點是流程簡潔、回應速度快，但需確保 Agent 的辨識精準度與標籤定義的一致性。第二種是進入系統後再依照權限進行控管，優點是權限管理較為集中與嚴謹，但需額外設計系統內的身份驗證機制。兩種方式的取捨待進一步評估後確認。

### 未來擴充方向

未來期望結合其他工具，讓系統在提供客戶資訊 summary 之後，能進一步協助使用者釐清問題原因，例如串接監控系統或日誌查詢工具，自動彙整相關紀錄。更進一步則可直接觸發後續處理步驟，例如自動建立工單、發送通知或執行對應的操作流程，減少人工介入的需求。

## 四、需求衝突

### 1. 備援機制 — OWNER 與 SRE

SRE 明確表示初期不考慮備援機制。OWNER 為避免資料獲取失敗，預計以固定排程下載檔案作為備用資料來源，本質上已是一種備援設計。兩個角色對備援的必要性認知相反，需在架構設計前對齊立場。

### 2. 系統定位 — 主管、OWNER 與 SRE

主管與 OWNER 將這套工具定位為中繼服務，核心職責是接收請求、查詢並整理客戶資訊後回傳，系統本身不是監控的主體。SRE 則將其視為需要主動監控的服務，關注 API 請求頻率、成功率與回應時間，目的是預防 AI 資源異常對服務造成影響。對系統定位的認知差異，會直接影響架構設計、維運責任與資源配置的方向。
