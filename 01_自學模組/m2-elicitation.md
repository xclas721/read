# M2:需求釐清 Elicitation(課前自學)

> 課程:**Spec-Driven 工程師 — AI Agent 時代的需求釐清訓練**
> 預計閱讀:30–45 分鐘
> 對應課堂場次:S2(W1:Stakeholder Discovery + 訪談)

---

## 學習目標

讀完這份你應該能:

1. 知道「問需求」為什麼是 AI agent 時代最不可外包的工作
2. 用**人事時地物(5W)**展開一份初步需求
3. 掌握訪談的基本動作:**問 → 聽 → 釐清 → 確認 → 追問**
4. 分得出「需求」與「假需求/解法」

---

## 為什麼這個模組

AI agent 寫程式很快,但它**完全不會幫你補你沒想到的事**。模糊需求 = 爛產出,而且 AI 會把錯的東西做得很漂亮,讓你以為對了。

兩大心法回顧:

- **探索**:需求不是一次問清楚就拿到的——它是一輪輪挖出來的
- **利益關係人**:AI 看不見人,漏一個 stakeholder,整個白做

這個模組教你**怎麼挖**。M3 教你怎麼把挖出來的東西結構化,M5 教你怎麼把結構化的東西寫成 AI 看得懂的規格。挖不到,後面都白搭。

## 核心(人人必修)

### 1. 需求有三層,你看到的常常只有第一層

| 層次 | 描述 | 例子 |
|---|---|---|
| 表面 | 對方說出口的 | 「我要一個查客戶資料的工具」 |
| 心裡 | 對方想的、但沒講清楚 | 「不只是查,還要看歷史互動,因為客訴常常重複發生」 |
| 隱性 | **對方沒想到、但會在意** | 「資料如果不夠新會誤判」「合規不允許看某些欄位」 |

第三層是 AI 完全看不見的——它只能讀你給它的。**你的工作就是挖第三層**。

### 2. 工具:人事時地物(5W)

不是用來填表格,是用來**逼自己想對方沒主動講的事**。

- **Who(人)**——誰會用?誰會被影響?誰會被批准?誰**不會主動上場但有 stake**?
- **What(事)**——他們想做什麼?怎麼做?產出長什麼樣?
- **When(時)**——什麼情境下做?平日?異常時?批量?即時?
- **Where(地)**——資料從哪來?到哪結束?跨系統嗎?
- **Why(為什麼)**——**這題最常被略過,也最重要**。為什麼要這個?現在沒有的話會怎樣?

> 練習一下:對你手上正在做的東西,5W 各問一個,看你能立刻答出幾個。沒答出的就是你的盲區。

### 3. 訪談五動作

| 動作 | 怎麼做 | 常見錯誤 |
|---|---|---|
| **問** | 開放式提問為主(「怎麼...?」「為什麼...?」)| 用是非題("你是不是想要 X?")——對方會點頭 |
| **聽** | 停頓,讓對方多講;不打斷 | 對方還在組織語言你就接話 |
| **釐清** | 不懂就問,不假裝懂 | 點頭應付,事後猜 |
| **確認** | 用自己的話覆述,讓對方點頭或更正 | 跳過確認,你以為的=對方說的 |
| **追問** | 每個答案背後都有沒講的細節:「然後呢?」「為什麼?」「如果...?」| 第一輪就想定案 |

### 4. 隱性需求三條挖掘線

- **例外情境**:正常情況沒問題,當 X 發生時呢?(查不到、權限不足、資料過舊、stakeholder 缺席...)
- **邊界**:資料來自哪?到哪結束?誰沒被涵蓋?
- **預設**:對方覺得「這還用講?」的事——那才是你需要確認的事

> 一個訣竅:當對方用「應該」「正常來說」「通常」這些詞,**那後面就是隱性需求的入口**。追下去。

### 5. 軟技能不是裝飾,是工具

- **傾聽**:停頓,讓對方說完——你會聽到本來不會講出來的東西
- **釐清**:重述 + 確認,把模糊變具體
- **同理**:理解對方的處境(為什麼他會這樣想?),不是同意他
- **不帶答案進訪談**:如果你已經知道答案,你就不是在訪談,是在驗證

### 6. 需求的三品質檢驗

寫下來的需求要過這三關:

- **完整**:該有的都有(5W 都答了嗎?例外情境呢?)
- **一致**:不自相矛盾(不同 stakeholder 講的對不上時,你發現了嗎?)
- **可驗證**:能寫出「達成 = 什麼長相」(不能驗證 = AI 做完你也不知道對不對)

## 進階延伸(有經驗者深化)

### 訪談反模式

| 反模式 | 長什麼樣 | 怎麼修 |
|---|---|---|
| 引導性提問 | 「你是不是想要 dropdown?」 | 改問「你想解決什麼問題?」 |
| 確認偏誤 | 只聽支持自己預設的答案 | 主動找反證 |
| 過早收斂 | 第一輪訪談就想定案 | 預期至少 2–3 輪迭代 |
| 解法導向 | 對方說「我要 X 功能」就直接記下 | 退一步問「為什麼需要 X?」 |

最後一條最重要:**對方給你的常常是解法,不是需求**。解法是他猜出來的答案,需求是他真正的問題。退一步問「為什麼」,你才挖得到真正的需求。

### 多 Stakeholder 衝突訪談

- 不同 stakeholder 對同一件事描述會不同——**這不是 bug,這是資訊**
- 不要在訪談中「解決」衝突,先各自挖完
- 衝突往往揭示了真正的需求:他們各自在意什麼

## 小範例:把人事時地物套到 workshop 題目

### 客戶資訊工具(W1–W3 主題)

| 5W | 你會問的問題(練習) |
|---|---|
| Who | 誰會用?手動查的痛點是?有誰不會主動上場但會被影響? |
| What | 要看到什麼?查不到時要回什麼? |
| When | 平日 / 客訴中 / 批次? |
| Where | 資料從哪來?多新? |
| Why | 為什麼這個工具現在才出現?之前是怎麼做的? |

> ⚠️ 這份範例**故意不給答案**。Workshop 開頭會讓你各組獨立猜 stakeholder map,然後比對誰找最多。先別偷看別人怎麼想。

### 郵件處理工具(W3 變體/buffer)

| 5W | 你會問的問題(練習) |
|---|---|

## Self-check

回頭看看你自己:

1. 你手上正在做的東西,5W 各問一個,你能**立刻答出**幾個?沒答出的就是你的盲區。
2. 上次跟同事討論需求時,有沒有「假裝聽懂了」就過去?事後是不是還是猜?
3. 你訪談時最常用的提問是「是不是 X?」(是非題),還是「為什麼 / 然後呢 / 如果...?」(開放式)?

## 預習問題(帶到 S2 課堂)

S2 一開始,各組會獨立寫 stakeholder map。讀完這份之後,先想一下:

1. 對「**客戶資訊工具**」這個題目,你心裡的 stakeholder 名單有哪些?——不限技術視角,任何「會用、會被影響、會被批准、會在意」的人都算。
2. 你能想到哪些「**stakeholder 不會主動講**」的隱性顧慮?(預設的、例外情境的、邊界的)
3. 如果你只能問 stakeholder **一個問題**,你會問什麼?為什麼是這個?

S2 開頭會比對各組的 map——**先別跟其他人對答案**,自己想過的版本最有價值。
