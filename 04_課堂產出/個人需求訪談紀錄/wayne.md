# 客戶資訊工具需求訪談紀錄 - Wayne

---

## 0. 整理原則

1. 只保留訪談中有提到的內容。
2. 不沿用原本文件中的 AI 式需求編號。
3. 不把未確認的事情寫成已決定。
4. 不額外加入產品判斷或技術方案。
5. 有疑問的地方放到「待釐清問題」。

---

## 1. 目前訪談整理方向

目前訪談內容大致可以分成幾個主軸：

| 主軸 | 內容 |
| --- | --- |
| 使用範圍 | 第一階段給部門內部使用，暫時不開放業務 |
| 權限控管 | 後台與 CLI 都需要權限控管 |
| 後台介面 | 需要後台介面，並支援 SSO |
| 客戶資料 | 資料以人工維護為主，第一階段以 DB 儲存 |
| AI Summary | 協助使用者快速理解客戶、角色、可信度與建議動作 |
| AI Agent | 可以參與資料操作，但操作後需要等待人工 Approve |
| 操作紀錄 | 後台操作需要有 Before / After 紀錄 |
| 歷史紀錄 | 客戶問題歷史紀錄需要留存，後續可提供 AI 學習或查找 |
| 備援資料 | 客戶資訊不可用時，需要定期下載或匯出客戶資訊作為備援，例如一週一次 |

## 2. 現場訪談內容整理

### 2.1 使用範圍與權限

| 項目 | 訪談內容 |
| --- | --- |
| 後台使用範圍 | 第一階段只開放部門內使用 |
| 業務是否使用 | 暫時不開放給業務 |
| 權限範圍 | 需要釐清新增、刪除、修改、查詢的權限 |
| 後台介面 | 需要後台介面，主要是方便人員閱讀與操作 |
| 人為修改審核 | 目前傾向用權限區分與操作歷史控管，不一定需要審核 |

### 2.2 AI Summary 與行動建議

| 項目 | 訪談內容 |
| --- | --- |
| 查不到客戶資訊時 | 提示應顯示在 AI Summary 區塊 |
| 查不到資料時的提醒 | 提醒使用者檢查舊資料，或建立新資料 |
| AI Summary 內容 | 需要包含客戶角色、職位、是否為技術人員 |
| 行動建議用途 | 協助值班、客服快速判斷是否需要處理，或只是正常告警 |
| 不同回應內容 | 目前傾向由 CLI 控制不同回應內容，發送前需要人工 review |

### 2.3 客戶資料維護

| 項目 | 訪談內容 |
| --- | --- |
| 資料可能延遲 | 客戶資料可能會延遲，因此需要定期確認是否正確 |
| 更新方式 | 透過排程定期詢問客戶資訊是否正確，有異動再人工登打更新 |
| 儲存方式 | 第一階段以 DB 為主 |
| RAG | 第一階段暫時沒有一定要使用 RAG 的需求 |
| 既有資料 | 需要支援既有客戶資料匯入與匯出 |
| 問題歷史紀錄 | 由另一個 Developer Site 管理留存，後續可提供 AI 學習或透過 RAG index 關鍵字查找 |

### 2.4 操作紀錄與 AI Agent

| 項目 | 訪談內容 |
| --- | --- |
| 後台操作紀錄 | 需要 |
| 操作紀錄呈現 | 使用 Before / After 方式呈現 |
| AI Agent 操作資料 | 可以開放 AI Agent 新增、刪除、修改 |
| AI Agent 審核 | AI Agent 做相關操作後，仍需人為在後台 Approve |

### 2.5 客戶標籤

| 項目 | 訪談內容 |
| --- | --- |
| 標籤定義 | 應先定義既定標籤清單 |
| 標籤權限 | 標籤清單定義後，再定義誰可以操作標籤 |

### 2.6 服務水平提升

| 項目 | 訪談內容 |
| --- | --- |
| 衡量方向 | 觀察原本人為處理的作業，是否逐步轉為 AI Agent 處理 |
| 量化方式 | 各階段應定義哪些事情逐漸由 AI 接管 |
