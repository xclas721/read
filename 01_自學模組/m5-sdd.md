# M5:SDD 規格 + Test-first 驗收(課前自學)

> 課程:**Spec-Driven 工程師 — AI Agent 時代的需求釐清訓練**
> 預計閱讀:45–60 分鐘(本模組稍長,本課最核心)
> 對應課堂場次:S4(W3:Spec 定稿 + AI 見面 demo)
> **附件**:[spec-template.md](spec-template.md) — W3 定稿時填用

---

## 學習目標

讀完這份你應該能:

1. 寫出 AI agent 可執行、**無歧義**的 Markdown spec
2. 先**定義驗收條件**,再讓 AI 實作——而不是做完才想怎麼驗
3. 建立「需求 ↔ 規格 ↔ 驗收」三層 traceability,需求改了知道哪裡要跟著動

---

## 為什麼這個模組

前面三站做完,你手上有:

- M2:訪談筆記(挖到的)
- M3:結構化需求清單 + 優先序
- M4:三視角的設計模型

接下來要做的事——把這些**寫成 AI agent 可以直接執行的形式**。這份文件就是「人對 AI 的介面」:寫得清楚,AI 做得對;寫得模糊,AI 會幫你補出一個漂亮的錯誤版本。

```
M2 訪 → M3 整理 → M4 設計 → M5 寫 spec + 驗收 → AI 實作 → 用驗收條件驗
```

兩大心法回顧:

- **探索**:第一版 spec 一定不對。AI 看了會反問、會補錯——這些都是進一步探索的訊號。**spec 是迭代產物,不是一次寫對**
- **利益關係人**:spec 寫得清楚 = 沒給 AI 機會「自行腦補」stakeholder 的意圖。每多寫清楚一句,就少一個 stakeholder 的盲區

## 核心(人人必修)

### 1. M5 的位置:寫給 AI 讀的文件

| 上一站(M2–M4)交給你 | M5 你要做的事 | 交給下一站(AI 實作) |
|---|---|---|
| 訪談筆記 + 需求清單 + 設計模型 | 結構化、無歧義、可驗證地寫成 spec + 驗收 | 一份 AI 可以直接讀、做、被驗收的 Markdown |

讀者是 AI——所以要結構化(段落清楚)、無歧義(用詞精準)、可驗證(驗收明確)。這三件事是 M5 的全部。

### 2. SDD 三性質:結構化、無歧義、可驗證

| 性質 | 壞例 | 好例 |
|---|---|---|
| 結構化 | 一段話講完所有需求 | 分節:目的 / Stakeholder / 用例 / 需求 / 設計 / 驗收 / 限制 |
| 無歧義 | 「系統要好用」、「介面要快」、「資料要安全」 | 「列表 P95 ≤ 800ms」、「客服角色看不到身分證後 4 碼」 |
| 可驗證 | 「客服可以查詢客戶」 | 「給定:已有客戶 A、客服 B 登入後輸入 A 的姓名;預期:回傳 A 的基本資料 + 近 6 個月互動」 |

每寫完一段,問自己這三個:「結構在哪段?用詞會不會被多種解讀?能不能寫驗收?」——三個都過,才往下寫。

### 3. Spec 的標準段落

SA 文件的架構保留,載體換成 Markdown。每份 spec 至少要有以下段落([spec-template.md](spec-template.md) 是現成範本):

| 段落 | 寫什麼 | 常見錯誤 |
|---|---|---|
| 目的 | 一句話:這個工具解決什麼問題、為誰 | 寫成「打造 XX 系統」(這是 what,不是 why) |
| Stakeholder | 會用 / 會影響 / 會批准 / 有 stake 的所有人 + 各自關心 | 漏掉「不會主動上場但有 stake」的角色(維運、合規、稽核) |
| 用例 / 使用情境 | Happy path + 至少 2 個例外情境 | 只寫 happy path |
| 功能需求 | 編號、可獨立驗收、標 MoSCoW | 寫成大段散文,無法各自引用 |
| 非功能需求 | 效能 / 安全 / 可用性 / 合規 | 完全省略——這是 AI 最容易補錯的部分 |
| 設計 | M4 的三視角(Mermaid 內嵌)+ 必要的文字補充 | 只貼圖、沒文字解釋 |
| 驗收條件 | 對每條功能需求,給可執行的驗收 | 「客服覺得好用」這類主觀條件 |
| 限制條件 | 技術、合規、時程、資源 | 只想到技術限制 |
| 開放問題 / TBD | 還沒釐清的、要回 stakeholder 問的 | 假裝都想清楚了(讀的人沒看到 TBD 反而以為 spec 完整) |

**開放問題段落很重要**——你寫了 TBD,讀的人(同事或 AI)知道「這裡我不確定」;你藏起來,讀的人會以為你想清楚了。

> **非功能需求(NFR)常被略寫,但 AI 最容易在這裡腦補**。寫 spec 時逐項問自己「這個工具需要嗎?」:
>
> | 類別 | 問自己 |
> |---|---|
> | 效能 | 多少資料量?多少並發?目標 P95 / P99 是多少? |
> | 安全 | 誰能看什麼?稽核留痕?敏感欄位遮蔽? |
> | 可用性 | 平日 / 異常時都能用?降級行為怎麼設計? |
> | 可維護性 | 誰之後要改?需要日誌 / 監控嗎? |
> | 合規 | 法規約束?產業規範?跨境傳輸? |
> | 可觀測性 | 出事誰收警示?怎麼定位問題? |
>
> 不是每條都要寫,但**每條都要問過**。沒問過就放空,就是「容易補錯」的源頭——AI 會自己挑一個版本實作,然後驗收時你才發現它跟你想的不一樣。

### 4. Test-first / ATDD 思維:驗收先想清楚

不是強制你跑 unit test、不是 TDD 教條——是**「AI 寫完你怎麼知道對」的思維前移**。

| 順序 | 結果 |
|---|---|
| 先寫驗收 → 再讓 AI 實作 → 用驗收驗 | 做完知道對不對;不對的地方明確 |
| 先讓 AI 寫 → 寫完再想怎麼驗 | 容易順著 AI 寫的內容定驗收(我們稱「事後合理化」);AI 寫錯你也驗收得了 |

W3 demo 你會看到:**AI 寫得很順、很漂亮,但不一定對**。沒有事先寫的驗收條件,你會懷疑「是不是我想錯了」——而不是「AI 做錯了」。

### 5. 驗收條件的寫法:可執行、可量化、邊界明確

| 模糊版本 | 結構化版本 |
|---|---|
| 系統要快 | 列表頁載入 P95 ≤ 800ms,1 萬筆資料情境 |
| 查不到要友善提示 | 查不到時回傳:① 友善訊息(如「查無此客戶,請確認拼字或試其他條件」)② 建議的下一步操作(改用電話 / Email 查) |
| 要安全 | 客服角色不可見:身分證後 4 碼、信用卡完整號碼;每次查詢寫 audit log(誰、何時、查誰) |
| 操作要直覺 | (不要這樣寫,直覺是主觀。如果你想要的是「3 次點擊內完成」,就這樣寫) |

**Given / When / Then 是常用格式**(也叫 BDD 風格):

> Given 客戶 A 已存在、客服 B 已登入
> When 客服 B 輸入「A 的姓名」並送出查詢
> Then 顯示 A 的基本資料 + 近 6 個月互動歷史,且寫入 audit log

這個格式 AI 讀得最直接——它知道前提、動作、預期結果。

### 6. Traceability:三層對齊

| 需求 ID | 規格段落 | 驗收 ID |
|---|---|---|
| R1 客服可依姓名查客戶 | §4.1 查詢功能 | T1.1 / T1.2 |
| R2 顯示近 6 個月互動歷史 | §4.2 互動歷史 | T2.1 / T2.2 |
| R3 敏感欄位權限分流 | §5.2 權限規則 | T3.1 / T3.2 / T3.3 |

寫完 spec 做一張這種表,**漏掉的會浮出來**:某個需求沒對應的驗收?某個驗收沒對應的需求(可能是 AI 補的或你多寫的)?——這張表就是 catch 機制。

### 7. 用詞警示(老 SA 的血淚教訓)

下面這些詞**會被不同人解讀成不同事**——AI 也會:

| 模糊詞 | 為什麼模糊 | 該怎麼寫 |
|---|---|---|
| 修改 / 新增 / 異動 / 配合 | 範圍不清、誰負責不清 | 明確寫:在 X 段新增 Y 欄位、由 Z 角色觸發 |
| 「優化」「改善」「強化」 | 改成多少算改了? | 給可量化目標:從 X 變到 Y |
| 「適當」「合理」「友善」 | 主觀 | 給可驗證條件或範例 |
| 「彈性」「擴充性」 | 對什麼彈性?未來什麼會變? | 列具體會變的維度 |

> 舊教材有句話:「**SA 文件應該為全團隊與客戶的共識**」——共識的反面是「每個人讀完心裡的版本都不一樣」。用詞模糊就是製造這種發散。

### 8. W3 AI demo 觀察點(預習用)

W3 中段講師會拿一份組的 spec 丟給 AI,全班一起看。**事先想好你要看什麼**:

- AI 會「**補了什麼你沒寫的**」?(可能補對、可能補錯)
- AI 會「**對什麼反問**」?(那就是你 spec 裡寫得模糊的地方)
- 用你的驗收條件去對 AI 的產出——**哪條 pass、哪條 fail**?

預習問題段會帶你具體思考。

## 進階延伸(有經驗者深化)

### 迭代下的 traceability 維護

需求變了,spec 跟驗收要跟著動。不維護的後果:

- 需求改了、spec 沒改 → AI 照舊 spec 做,做了一個你已經不要的東西
- spec 改了、驗收沒改 → AI 做新版,你用舊驗收一驗永遠不過
- 三者都改了、traceability 表沒改 → 半年後沒人記得這條怎麼來的

維護成本不高,但要養習慣:**改任一層,順手把另兩層 + traceability 表一起改**。版控配 git commit 用,每次改有跡可循。

### 規格給 AI 讀,跟給人讀有什麼不同

點到為止——這題 W3 demo 後會自己有體會,先不深入:

- AI 讀得**比較字面**:你寫「快」它真的不知道多快,人會猜「800ms 吧」,AI 不會
- AI 讀得**比較直接**:你寫「客服查客戶」,人會自動補「找不到的話顯示提示」,AI 會直接做完不處理找不到的 case
- AI 讀得**比較全**:它不會跳過 TBD、不會略過你不想處理的段落——你寫什麼它做什麼

**結論很簡單**:寫給 AI 讀的 spec,等於把「給人讀時你預期對方會自動補的部分」全部寫出來。一開始覺得囉嗦,但 AI 做出來會少很多坑。

### 何時需要圖、何時純文字

跟 M4 同原則:**3 個以上元素 + 有關係**才畫,線性流程列點就好。M5 的圖通常重用 M4 的——不要重畫,直接複製 Mermaid。

## 小範例:迷你 spec(個人筆記工具)

故意**不用** workshop 題目「客戶資訊工具」——避免洩漏 W3 答案。看這份感受 spec 長相:

````markdown
# 個人筆記工具 Spec

## 目的
讓使用者快速記下零散想法,日後能按主題找回。

## Stakeholder
| 角色 | 主要關心 |
|---|---|
| 使用者 | 記得快、找得到 |
| 未來接手者(同事/AI) | spec 夠不夠他照做 |

## 用例
1. Happy path:使用者按熱鍵 → 跳出輸入框 → 寫一句話 + 標籤 → 存
2. 例外 1:沒網路 → 存在本地,連線後同步
3. 例外 2:標籤拼錯 → 自動建議近似已有標籤

## 功能需求
- R1 (Must) 熱鍵呼叫輸入框
- R2 (Must) 一句話 + 一個標籤的最小輸入
- R3 (Should) 按標籤搜尋

## 驗收條件
- T1.1 Given 應用在背景 / When 按下熱鍵 Cmd+Shift+N / Then 200ms 內彈出輸入框
- T2.1 Given 輸入框打開 / When 輸入「memo + #idea」按 Enter / Then 紀錄存檔,輸入框關閉
- T3.1 Given 已有 10 筆筆記 / When 在搜尋輸入「#idea」 / Then 列出所有含 #idea 標籤的筆記
````

**注意**:這份 spec 故意不完整(沒寫設計、沒寫非功能需求)——只是讓你看「結構化 + Given/When/Then」長什麼樣。W3 你要做的是**完整版**。

## Self-check

回頭看看你自己:

1. 你過去寫的設計文件,如果丟給 AI,**AI 會在哪裡卡住**?——卡住通常是**探索不足**(該問的還沒問)還是**漏 stakeholder**(整個視角沒納入 spec)?
2. 你定的驗收條件,有多少是「**可執行的測試**」、多少是「**主觀感覺**」?
3. 需求若改了,你的 spec 跟驗收條件怎麼**同步**?——沒同步機制,代表 traceability 還沒建立。

## 預習問題(帶到 S4 / W3)

W3 開始前:

1. 你的組 W2 累積的 spec 草稿,對照本模組的「Spec 標準段落」表,**缺哪幾段**?
2. 挑你 spec 中**最模糊**的一條需求,試著改寫成「結構化 + 可驗證」的版本(用 Given/When/Then)。
3. W3 中段 AI demo 時,如果 AI 對你的 spec 反問了**一個問題**,你會希望它問什麼?——能說出來,代表你已經知道你的 spec 哪裡虛。
