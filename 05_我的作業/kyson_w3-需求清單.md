## 基本資料

- 姓名 / 組別：Kyson / SRE 團隊
- 工具：**客戶資訊工具**（AI Agent 生態的基礎識別層，優先開發）

## 2. 非功能需求

> 效能類**必附規模假設**。寫不出規模的先別列。

- **效能**：單筆查詢 < 1s；端到端回應 1 分鐘內可接受。
  - 規模假設：客戶約 **300 筆**，請求型態為**觸發式低併發**（壓測估 1000 同時仍可撐）。
  - ⛔ 結論：Queue／排隊機制**不做**——量化規模後屬假議題（A組衝突1 裁決 WON'T）。
- **安全**：
  - 個資（email／手機／姓名）依個資法 **AES 加密、不明碼**（C組衝突7）。
  - 查詢全程留 Audit Log（人工進 DB／API 寄 log）。
  - Tool ↔ AI Agent 同一 domain 內通訊，**不需額外加密**（Kyson 訪談）。
  - API 需限流、呼叫次數與成功率監控、AI token 用量控制與告警（Ray 訪談）。
- **可用性**：
  - 工具掛掉須能以人工備份立即查（Chloe）。
  - 為平衡成本，可接受**冷啟動／排程喚醒／彈性擴展**達成 SLA（C組衝突1，Owner 與主管均同意）。
  - 規模假設：客戶約 300 筆，非核心交易系統。
- **合規**：
  - 依個資法（AES 加密）。
  - ISO-27001：**公司目前尚未導入**；若導入，資料流程異動須留紀錄，必要時須 approve（C組衝突7）。
  - PCI：涉資安操作須可追責（執行者／approve／主管）。

## 3. 開放問題 / 待 escalate

> 沒拍板的、牽涉風險的、兩案並陳的——全列，並標「需向誰確認」。

- [ ] **domain 自動歸戶 vs CC 人工防詐騙**——資安取捨，已先做寄件人 domain；CC 人檢查是否要做、做到什麼程度 — 需先向主管講風險再決定。**（資安風險，待 escalate）**
- [ ] **AI ↔ Tool 對接流程方案未定**：方案 A（訊息→AI→客戶資訊工具→AI→其他工具）vs 方案 B（訊息→客戶分層→客戶資訊工具→AI→其他工具）— 待 AI Agent 整體流程定案。
- [ ] **部署平台 AWS vs GCP**：SRE 偏 AWS（熟悉、整合好），主管要求「便宜優先」— SRE 將列 AWS／GCP 價格交主管審核。**（待 escalate）**
- [ ] **ISO-27001 導入時程與 approve 流程範圍**：導入後資料異動留痕與 approve 的具體門檻 — 待主管／資安拍板。**（待 escalate）**
- [ ] **歷史紀錄存放位置**：放 Developer Site vs 與客戶資訊工具放一起（利於 AI 追蹤客戶轉交脈絡）— 待 AI Agent 流程定案。
- [ ] **服務常駐 vs 冷啟動的最終 SLA 數字**：在最低成本下滿足 SLA 的具體秒數 — 待壓測結果與主管確認。
- [ ] **後台 UI 是否要做**：主管無偏好、Owner 偏 CLI — 初期 CLI，後續視需求評估。
- [x] **AI 自動改防火牆／加 IP 白名單**：⛔ WON'T——違反 PCI、自動化無人可追責（A組衝突6，已裁決不做，列此備查）。
