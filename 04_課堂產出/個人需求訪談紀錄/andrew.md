# 客戶資訊工具 需求訪談紀錄

## 一、主管訪談需求整理

### 訪談對象
- 角色：主管

### 需求整理

#### 客戶識別與服務分類
- 透過 Email Domain、通訊頻道等資訊識別客戶身分。
- 自動判斷客戶所使用的服務（ACS、3DSS、veriid 等）。
- 未來由 AI 自動完成分類，降低人工判斷成本。

#### 客戶資料管理
##### 必填資料
- 客戶姓名
- 角色／職責（可複數標籤）
- 使用服務（ACS、3DSS、veriid）
- SLA

##### 選填資料
- Email
- WhatsApp 等通訊工具資訊

##### 客戶資料結構
```text
公司
├─ 服務
│   ├─ SLA
│   └─ 聯絡人
│       ├─ 姓名
│       ├─ 角色
│       └─ 通訊方式
```

- 需記錄 HiTRUST 內部業務聯絡窗口。
- 提供資料匯入、匯出與批次修改功能。

#### SLA 管理
- 記錄每位客戶、每項服務對應的 SLA。
- 支援公版 SLA 與客製 SLA。
- 多個 SLA 存在時，以最嚴格條件為準。
- 未來可結合告警系統判斷回覆時限。

#### 歷史問題知識庫
- 建立獨立的歷史問題資料庫。
- 保存過往問題、處理方式與 QA 紀錄。
- 提供 AI Agent 查詢與回覆建議。
- 未來可導入 RAG 或向量資料庫。

#### 權限與資料治理
- 初期僅提供內部使用。
- 系統需支援 SSO。
- AI Agent 以只讀權限為主。
- 新增、修改、刪除資料需透過後台人工審核。
- 需考量資料權限控管與敏感資訊存取限制。

#### 客戶資料維護機制
- 定期檢查客戶資料是否過久未更新。
- 依據 Last Update 時間進行提醒。
- 每季整理客戶資訊並通知相關窗口確認資料正確性。

#### 預期效益
- 降低人工查找資料時間。
- 縮短客戶問題回覆時間。
- 提升告警處理效率。
- 累積組織知識庫。

## 二、SRE 訪談需求整理

### 訪談對象
- 角色：SRE

### 需求整理

#### 資安與法規遵循
- 考量個資與 ISO 相關規範。
- 儲存資料需評估加解密機制。
- 支援白名單管理機制。
- 對外客戶相關操作以 IP 白名單與資安設定為主。

#### 系統架構與高可用性
- 客戶資訊工具作為獨立服務建置。
- AI Agent 部署於 GCP。
- 採用 Kubernetes 提升容錯能力。
- 資料庫需具備備援與容錯能力。
- 兼顧高可用性與成本控制。

#### 監控與維運
##### API 監控
- API 呼叫次數。
- API 成功率與失敗率。
- Response Time。

##### AI Agent 監控
- Token 使用量監控。
- Token 達警戒值通知維護人員。

##### 系統監控
- Nginx Log。
- AP Log。
- 4xx、5xx 錯誤監控。
- Log 關聯分析。

#### AI Agent 自動化維運
- AI 分析異常並提供建議。
- 人工審核後執行。
- 逐步導入低風險自動化操作。
- 未來可支援自動擴容與基礎設施管理。

#### 告警與問題排查
- AI 協助分析告警原因。
- 自動整合 Nginx 與 AP Log。
- 快速定位問題來源與處理方向。

#### 效能與容量規劃
- 上線前需進行壓力測試。
- 根據系統定位規劃資源成本。
- 次要服務可接受較長回應時間。

#### Queue 與流量管理
- 建立 Queue 緩衝機制。
- 大量流量時先排隊再擴容。
- 設定各元件 Priority。

#### SLA 與服務降級機制
- Queue 等待時間需符合 SLA。
- 以最嚴格 SLA 為基準。
- 超過 SLA 時可由 AI 直接產生預設回覆。

#### 預期效益
- 提升系統穩定性與容錯能力。
- 降低人工維運成本。
- 縮短告警處理時間。
- 在 SLA、效能與成本間取得平衡。

## 三、產品專案負責人訪談需求整理

### 訪談對象
- 角色：專案負責人

### 需求整理

#### 客戶資訊快速查詢
- 快速查詢客戶資訊。
- 快速查詢客戶聯絡 Email。
- 辨識客戶使用的服務（ACS、3DSS）。
- 根據寄件人資訊識別客戶身分。
- 提供唯一且正確的資料來源。

#### 客戶溝通與回覆輔助
- 協助判斷客戶問題類型。
- 協助掌握 SLA。
- 保存歷史回覆紀錄與文件。
- 標記需謹慎回覆的問題。
- 建立統一官方回覆內容。

#### 標籤管理機制
##### 客戶標籤
- 發卡行名稱。
- 客戶公司名稱。

##### 服務標籤
- ACS。
- 3DSS。
- 其他服務。

##### 客戶角色標籤
- PG。
- 技術窗口。
- 業務窗口。
- 緊急聯絡人。

##### 問題標籤
- 交易問題。
- 系統問題。
- 公告通知。
- 維護通知。

#### 訊息分類與回覆模板
- 自動分類 Email 與訊息內容。
- 套用標準回覆模板。
- 公告信、維護信、交易查詢皆有固定格式。

#### 權限管理
- 查詢資料以只讀為主。
- 權限主要區分新增、修改、刪除。
- 一般查詢不需額外區分權限。

#### 客戶資料維護
- 目前維持人工更新。
- AI Agent 可協助更新與提醒。
- 避免多系統資料同步問題。
- 以服務工具作為主要資料來源。

#### 客戶身份驗證
- AI 協助判斷聯繫者真實性。
- 無法確認時由人工審查。
- 可透過歷史郵件與 CC 紀錄驗證。

#### 功能開發優先順序
1. 客戶資訊資料庫。
2. 客戶資訊與 Email 查詢。
3. 訊息分類與回覆建議。
4. AI 自動化分析與回覆。

#### 備援與資料保存
- 建立資料備份機制。
- 備份至 GitLab Repository。
- 使用 Markdown 保存重要資訊。
- 建立自動化備份流程。

#### 客戶資訊維護週期
- 每半年檢查與更新一次客戶資訊。

#### 預期效益
- 快速找到正確聯絡資訊。
- 縮短客戶回覆時間。
- 提升回覆一致性。
- 建立標準化客戶服務流程。

## 四、需求衝突分析

### 衝突點：SLA 與 Queue 機制

#### SRE

希望：

* Queue 消化大量請求

#### 專案負責人

希望：

* 客戶能快速得到回覆

#### 衝突原因

排隊可能導致超過 SLA。

#### 建議處理方式

建立服務降級機制：

* Queue 超時
* 無法取得資料

則：

* AI 直接回覆標準訊息
* 滿足 SLA 要求
